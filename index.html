Infrastructure in front of every model call

LLM Gateway with Rate Limiting and Fallback Routing

One gateway sits in front of every LLM call an organization makes. It enforces per team rate limits and budgets, fails over to a healthy provider during an outage, and traces every request end to end.

Why this system exists

The operational problem it solves, and why teams keep rebuilding it by hand until someone gives them the tooling.

The problem every growing team hits

Once more than one team calls language models, nobody can answer who spent what, which provider is failing right now, or how to stop one team burning the whole month of budget in an afternoon. Teams rebuild this by hand, badly, every time.

Why it lands interviews

This is pure infrastructure engineering applied to models, exactly the strength a mid level backend engineer already has. It leads with reliability numbers and operational control, not with a chatbot.

What it actually guarantees

Sub millisecond rate limit checks, a hard budget ceiling per team, automatic retry then failover on retryable errors, and a circuit breaker that stops hammering a dead provider.

Why the design holds under load

Token buckets refill on read so they need no background timer, fallback chains are defined per model tier so an option always exists, and every hop emits a span plus Prometheus metrics.

The exact workflow

Every request or task travels this path. The strip lights up in sequence so you can see the order the stages fire in.

01
Authenticate

The team API key is validated and its plan, allowed models, and limits are loaded.

02
Rate limit

Token buckets for requests and tokens are checked atomically; batch traffic yields to realtime.

03
Budget gate

Cost so far is compared to the daily cap; a warning fires at 80 percent, a block at 100.

04
Route and retry

The primary provider is called with exponential backoff on retryable errors.

05
Fallback

If retries exhaust or the breaker is open, the next provider in the tier chain serves the request.

06
Observe

Tokens, cost, latency, and provider are recorded as a span and exported as metrics.

How it works in depth

The mechanisms behind each stage, and the design decisions that make the system behave well under real load.

1
Unified provider interface

OpenAI, Anthropic, and Ollama sit behind one request and response shape, so the caller never knows or cares which provider served the call. A deterministic simulation mode runs the whole gateway with no API keys.

2
Token bucket rate limiting

Per team buckets for requests per minute and tokens per minute refill on read, the same logic a Redis Lua script runs in production. A batch priority keeps headroom so a data job never starves a user facing request.

3
Budget enforcement

Each request's cost is computed from real per token pricing and charged against a daily ceiling. Eighty percent raises a warning, one hundred blocks with a clear message pointing at the admin API.

4
Retry, fallback, and circuit breaker

Retryable errors retry the primary with backoff, then walk the tier fallback chain. After five failures in the window the breaker opens, traffic shifts entirely, and a single probe recloses it on recovery.

5
Health checking

Every provider is probed on a rolling window so its status is healthy, degraded, or down before a real request needs it, with the transition history kept for incident review.

6
Tracing and metrics

Spans cover auth, limit, budget, provider selection, and delivery. The metrics endpoint renders Prometheus text with requests, errors, latency percentiles, cost per team, and fallback rate.

How to run it

Runs fully offline with deterministic providers, so the demo and the tests reproduce byte for byte. Add API keys to switch to live models.

# offline load test and failover demo, no API keys
pip install -r requirements.txt
python demo.py
# -> mixed traffic, simulated outage, fallback, breaker open then recover

# serve the gateway
uvicorn gateway.api:app --app-dir src --port 8000
curl -s localhost:8000/v1/completions \
  -H "Authorization: Bearer gw-search-7f3a" \
  -d '{"model": "gpt-4o", "prompt": "Summarize this incident"}'
